# Lap 1 — Talk session notes (captured live)

Raw notes from the speaker, plus how we apply each to OUR arena agent. Design only — not built yet.

## Why AI agents fail
- Poor planning
- Hallucinated actions
- Context overload
- Weak tool usage
- No memory
- No eval

## Building blocks of good AI agents
Model · Memory · Planning · Tools · Knowledge · Eval · Feedback

## Planning
- **ReAct framework** for planning (reason + act interleaved). ← our `agent/agent.py` already uses a ReAct loop.

## Good practices (from the talk)
1. Optimize for **business outcome, not model intelligence**.
2. Design **explicit workflows**.
3. **Separate thinking from acting** — e.g. have it write a program to read lines inside files rather than eyeballing them.
4. **Tool quality matters more than model quality.**
5. Use **structured planning**.
6. Use a **mix of memory types**.
7. **Retrieval is better than bigger context.**
8. **Add reflection from day 1.**
9. **Enforce guardrails at the code level** — hardcode business logic instead of just telling the agent in the prompt.

## How we apply this to the Arena agent (mapping advice → action)

| Talk point | What it means for us | Concrete action (later) |
|---|---|---|
| No eval / Add reflection from day 1 | Already our #1 decision. | Draft→**reflect/eval**→submit gate; loop until PASS or `skip_task`. |
| Weak tool usage / Tool quality > model quality | Arena penalizes lucky guesses; tools = the points. | Build solid `web_search`, `calculate`, `run_python` helpers; route each task category through its right tool. |
| Hallucinated actions | Wrong submit = one-shot waste + ban risk. | Reflection step verifies the answer with a tool (run_python/web_search) before submit. |
| Separate thinking from acting | Don't answer math/code from the model's head. | For code/math tasks, generate code → **execute it** → submit the verified result, not a guess. |
| Enforce guardrails at code level | Don't rely on prompt-only rules. | Hardcode: step caps, repeat-action guard (already have), one-shot-submit protection, mandatory-tool-for-category check, UUIDv4 generation, JWT-expiry guard. |
| Structured / explicit planning | Stable, debuggable loop. | Explicit state machine: register → get_task → plan → act(tool) → reflect → submit/skip. Keep tool order stable (prompt-cache). |
| Optimize for business outcome | "Business outcome" here = **leaderboard score / Neural Efficiency**, not pretty reasoning. | Tune toward the actual scoring metric: correctness to clear ≥70, then token efficiency. |
| Mix memory types | Arena tasks are sticky & we level up over time. | Short-term: conversation/session (per our loop). Longer-term: remember per-category what tool/approach scored well; track level history (RunState already does some of this). |
| Retrieval > bigger context | Keep prompts tight (also helps Neural Efficiency). | Pull only the relevant task description + needed snippets into context; don't dump everything. Use web_search/RAG for Context-Awareness tasks instead of stuffing context. |
| Context overload | Token cost hurts Neural Efficiency + degrades quality. | Effort-gate by task difficulty; trim system prompt; summarize prior turns rather than carry full history. |

## Net: our agent's required pieces (talk's 7 building blocks → us)
- **Model:** claude-opus-4-8 (effort-gated).
- **Memory:** session state + per-category what-worked memory + level history.
- **Planning:** explicit ReAct state machine.
- **Tools:** web_search, calculate, run_python + the 4 arena MCP tools — high quality, the point-getters.
- **Knowledge:** retrieval (web_search/RAG) over big context.
- **Eval:** reflection/verification gate before every submit. ← non-negotiable.
- **Feedback:** read `submit_task` score + LEVEL_UP signal; feed into the what-worked memory.
