### DESIGN DECISION — add a self-EVALUATION step before every submit (don't build yet)
- The arena loop is get → (solve) → submit | skip. We add an **evaluation/verification gate
  between solve and submit** to raise accuracy, because:
  - `submit_task` is **one-shot per task** (can't resubmit) → a wrong submit is wasted points + risks bans.
  - Score ≥ 70 gates level-up; we want to clear that confidently, not gamble.
- Plan (to implement later): after the agent drafts an answer, run a separate **evaluator pass**
  (LLM critic / self-check, ideally a fresh context or stricter prompt) that scores the draft
  against the task's criteria and returns PASS / REVISE. Loop draft→eval up to N times, then:
  - eval says good → `submit_task`
  - eval keeps failing / task looks impossible → `skip_task` (no penalty) instead of a doomed submit.
- Use helper tools INSIDE the eval where they verify correctness: `run_python` to execute/check
  code & math, `web_search` to fact-check factual answers (tutorial says lookup turns 65→85).
- Keep it cheap: gate eval depth by task difficulty/level; tighter `effort` on easy tasks so the
  evaluation cost doesn't tank Neural Efficiency (token-efficiency score).
- Reuse existing loop guards (MAX_STEPS + repeat-action) so the draft↔eval loop can't spin forever.

### SCORING RULE — must SHOW tool use or get penalized (don't build yet)
- **If we don't use the right tools, the arena penalizes us — it assumes the answer was a lucky guess.**
  A correct final answer alone is NOT enough; the agent has to *demonstrably call the appropriate
  tool* for the task (e.g. `calculate`/`run_python` for math & code, `web_search` for factual lookups).
- Implication: this is the opposite of pure token-minimization. Skipping a tool to save tokens can
  cost more than it saves if it triggers the lucky-guess penalty. Balance: use the tool the task
  actually warrants, every time — don't answer from the model's head on tool-warranted tasks.
- Build implications (later):
  - Map each task category → its expected tool, and make the agent route through that tool before submitting.
  - The eval/verification gate should also CHECK "did we use the right tool for this task type?"
    and force a tool call if the draft skipped it.
  - Make tool calls visible in whatever trace/metadata `submit_task` sends (tutorial passes `metadata`).

### HOW TO COMPETE — stage-by-stage playbook (from docs §02/§03)
Compete loop (§02): 1) INITIALIZE SESSION (open SSE to MCP endpoint) → 2) IDENTITY VERIFICATION
(pass Platform UID + JWT in init payload; unauthorized agents are *instantly purged*) → 3) TASK
EXECUTION LOOP (`get_tasks` → solve → `submit_task` with a valid UUIDv4 executionId) → 4)
EFFICIENCY OPTIMIZATION (lower token cost at high score = higher Neural Efficiency → Sigma board).

Stage gates (§03) and what each demands of us:
- **Stage 1 — COLD BOOT:** registration + protocol handshake; they verify tech stack & architecture.
  → Goal: connect cleanly and `register_agent` succeeds (LinkedIn+GitHub URLs ready). Don't fumble the handshake.
- **Stage 2 — DEPLOYMENT TIER:** Level 1 challenges; **must demonstrate functional correctness or
  get banned.** → Goal: prioritize CORRECTNESS over cleverness/speed here. Use the eval gate +
  right-tool rule from the start so we never submit a wrong/lucky-guess answer and get banned out early.
- **Stage 3 — EVOLUTIONARY CYCLE:** scaling performance + token management; **promotion to higher
  gates starts at Score ≥ 70.** → Goal: clear ≥70 reliably AND start trimming token cost (effort-gating)
  so Neural Efficiency climbs. This is where correctness + efficiency must both be good.
- **Stage 4 — APEX BENCHMARKING:** level-locked complex tasks; max point yield + leaderboard dominance.
  → Goal: spend more reasoning/tools on these high-value tasks; this is where the ranking is won.

Operating principles across stages:
- **Survival first (Stages 1–2): never get banned.** A ban ends the run — correctness + the eval
  gate matter more than token savings early.
- **Efficiency second (Stages 3–4):** once correctness is locked, push Neural Efficiency down on
  easy tasks and spend big only on APEX tasks.
- `get_tasks` is **sticky** (same task until skip/submit) and `submit_task` is **one-shot** → use
  `skip_task` (no penalty) rather than burn the single submit on a task we can't pass.
- Docs confirm `submit_task` DOES take `executionId: UUIDv4` (matches docs, not the tutorial's
  shorter shape) — generate a fresh UUIDv4 per submit.
